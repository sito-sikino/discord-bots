# Discord Bots プロジェクト - 要件定義書

## 📋 プロジェクト概要

### プロジェクト名
**「Discord Bots」** - マルチボットLLMシステム

### 目的
Gemini 2.0 Flash APIを活用した**3つの独立したDiscord Bot**による協調型対話システムの構築

### 重要な前提
- **3つの独立したDiscordボット**がそれぞれ異なる役割を担当
- **統合プロセス型実装**：1つのプロセス内で3つの独立したDiscordクライアントを管理
- **Discord独立性**：各ボットが独立したトークン・アイコン・名前・個性を持つ
- 1つのDiscordサーバー内で協調動作
- **ユーザーは一人のみ**（個人利用想定）
- **共有知識システム**（Memory + Obsidian Vault）を全ボットが共有
- **レート制限対応**：Gemini API呼び出しを最大2回のシーケンシャルアクセスに制限

## 🤖 システム構成

### LangGraphベース統合エージェント設計
```
Single Python Process + LangGraph Workflow
├── @Spectra Client (TOKEN_SPECTRA) - 基底エージェント兼一般対話ボット
├── @LynQ Client (TOKEN_LYNQ) - 論理分析専門（ワークフロー制御）
└── @Paz Client (TOKEN_PAZ) - 創作専門（ワークフロー制御）

LangGraph StateGraph処理フロー
START → analyze_node → routing_decision → agent_nodes → END
                     ↓
        ┌─────────────┬─────────────┬─────────────┬─────────────┐
        │  Pattern 1  │  Pattern 2  │  Pattern 3  │  Pattern 4  │
        │ Spectra直接 │ LynQ委譲    │ Paz委譲     │ Multi協調   │
        └─────────────┴─────────────┴─────────────┴─────────────┘
        ↓             ↓             ↓             ↓
    spectra_respond  lynq_respond   paz_respond   multi_coord
        ↓             ↓             ↓             ↓
        END           END           END           END

最適化ポイント（LangGraph版）
├── StateGraphによる明確なワークフロー定義
├── TypedDictによる構造化状態管理
├── 条件付きエッジによる自動ルーティング
├── ChatGoogleGenerativeAI公式統合
└── LangChain標準パターン準拠
```

### システム構成の特徴（LangGraph版）
- **Discord独立性**: 各ボットが独立したDiscord接続・トークン・アイコン・名前を持つ
- **プロセス統合**: 1つのPythonプロセスで3つのDiscordクライアント + LangGraphワークフロー
- **StateGraph制御**: LangGraphのStateGraphによる明確で可視化されたワークフロー
- **TypedDict状態**: 構造化された状態管理（AgentState）とoperator.addによるメッセージ蓄積
- **条件付きルーティング**: add_conditional_edgesによる動的エージェント選択
- **LangChain準拠**: ChatGoogleGenerativeAI + 公式パターンによる標準化
- **エラー耐性**: 統一されたエラーハンドリングとグレースフルな縮退機能
- **メンテナンス性**: 公式パターン準拠による高い保守性と拡張性

## 🔄 LangGraphアーキテクチャ

### StateGraphワークフロー動作パターン

#### パターン1: Spectra自己応答（一般対話）
```
START → analyze_node → routing="spectra" → spectra_respond_node → END
ユーザーメッセージ → 深層分析（1回目LLM） → Spectra直接応答（2回目LLM）
総LLM呼び出し: 2回
```

#### パターン2: 専門ボット直接委譲（効率化）
```
START → analyze_node → routing="lynq/paz" → lynq/paz_respond_node → END
ユーザーメッセージ → 深層分析（1回目LLM） → 専門エージェント応答（2回目LLM）
- StateGraph内で完結、中間Discord投稿なし
- 総LLM呼び出し: 2回
```

#### パターン3: マルチエージェント協調（複合要求）
```
START → analyze_node → routing="multi" → multi_coordinate_node → END
ユーザーメッセージ → 深層分析（1回目LLM） → 協調調整メッセージ → 複数エージェント順次実行
- LangGraph内での順次実行制御
- 総LLM呼び出し: 3-4回（複合要求のため許容）
```


### Spectraの判断基準（LLMによる知的分析）
- **深層理解**: 文脈・意図・感情・背景の総合分析
- **最適性判断**: どのボットが最も価値ある応答を提供できるか
- **協調判定**: 単独応答 vs 複数ボット協調が必要か
- **メタ視点**: システム全体として最良の結果を目指す

### LLM調整システム（技術的課題の解決）

#### Gemini API Coordinator
- **APIキューイング**: 3つのボットからのリクエストをキューで管理
- **順次処理**: 同時リクエストを避け、順番に処理
- **応答キャッシュ**: 同一内容の重複リクエストを回避
- **タイムアウト管理**: 適切な待機時間を設定

#### 効率化された応答制御メカニズム
1. **単独受信**：Spectraのみがメッセージを受信
2. **一元判定**：Spectraが高度なLLM分析で応答戦略を決定
3. **直接委譲**：専門ボットへの内部シグナルによる即座の起動
4. **重複完全回避**：単一制御点により重複応答ゼロ
5. **協調制御**：複合要求時のみSequential実行

#### 軽量技術実装
- **AsyncIOキュー**: Python標準ライブラリによる内部シグナリング
- **ファイルベースMemory**: 軽量な履歴共有
- **レート制限対応**: 最大2回のシーケンシャルLLMアクセス
- **ゼロ外部依存**: Redis不要、データベース不要

### Obsidian MCP連携

#### 共通知識ベース（重要な共有リソース）
- **アクセス方法**：全ボットがObsidian MCP Server経由でアクセス
- **アクセス頻度**：**任意のタイミングで情報取得・引用が可能**
- **検索タイミング**：
  - 関連キーワード検出時の自動検索
  - ボットの判断による能動的な知識検索
  - 過去の資料や記録が必要な場面での参照
- **活用方法**：各ボットの特性に応じて知識を活用
- **共有性**：Memoryと同様に、全ボットが同じVaultを共有

#### MCP統合仕様
- **接続方式**：Model Context Protocol経由
- **対象リポジトリ**：https://github.com/MarkusPfundstein/mcp-obsidian
- **機能**：
  - vault内容の取得
  - ファイル内容の引用
  - 関連情報の検索
  - **任意のノートやファイルへの直接アクセス**
  - **知識の横断的な活用**

### LangGraphワークフロー実行フロー
```
Discord Message (Spectra単独受信)
↓
build_agent_state (AgentState構築)
↓
START → analyze_node (深層分析 + メタ判断 - 1回目LLM)
↓
routing_decision (条件付きエッジ)
↓
┌─────────────┬─────────────┬─────────────┬─────────────┐
│  Pattern 1  │  Pattern 2  │  Pattern 3  │  Pattern 4  │
│"spectra"    │"lynq"       │"paz"        │"multi"      │
└─────────────┴─────────────┴─────────────┴─────────────┘
↓             ↓             ↓             ↓
spectra_      lynq_         paz_          multi_
respond_node  respond_node  respond_node  coordinate_node
(2回目LLM)    (2回目LLM)    (2回目LLM)    (調整メッセージ)
↓             ↓             ↓             ↓
END           END           END           END
↓             ↓             ↓             ↓
Discord Response (AgentState.messagesから最新取得)
↓             ↓             ↓             ↓
Conversation Memory Update (統一的な履歴管理)
```

## 🎯 基底エージェント設計の詳細説明（LLM知的判断を活用）

### 設計の根本思想
**問題**: 軽量判定（キーワードベース）では**LLMの知的判断力を無駄にしてしまう**

**解決**: **Spectraが基底エージェントとしてLLMの知的分析力を最大活用**

### 核心的なアプローチ
```
Spectra = 基底エージェント
↓
全メッセージをLLMで高度分析
↓
「誰が最適か」をメタ的に判断
↓
自身応答 OR 専門ボットへバトンタッチ
```

### Spectraの知的判断プロセス

#### ステップ1: 深層理解（LLM使用）
```python
def deep_analysis(message, context, history):
    analysis = gemini_api.analyze({
        "message": message,
        "context": context,
        "conversation_history": history,
        "task": "深層的な意図・感情・背景の分析"
    })
    return analysis
```

#### ステップ2: 最適性判断（LLM使用）
```python
def meta_decision(analysis):
    decision = gemini_api.decide({
        "analysis": analysis,
        "agents": {
            "spectra": "統合・ファシリテーション・一般対話",
            "lynq": "論理分析・問題解決・客観的思考",
            "paz": "創作・アイデア・発想・インスピレーション"
        },
        "task": "最も価値ある応答を提供できるエージェントを選択"
    })
    return decision
```

#### ステップ3: 実行戦略
- **Spectra応答**: 統合的・ファシリテーション的な応答が最適
- **バトンタッチ**: 専門性が求められる場合
- **協調応答**: 複数の視点が必要な複合要求

### 設計のメリット

#### 🧠 知的活用
- **LLM最大活用**: 軽量判定ではなく、高度な文脈理解
- **メタ認知**: 「誰が応答すべきか」の客観的判断
- **価値最大化**: 各状況で最も価値ある応答を提供

#### 🤖 ボット独立性（維持）
- **物理独立**: 各ボットが独立したDiscord接続・プロセス・Token
- **キャラクター保持**: 各ボットが独自の@名前・アイコン・性格
- **専門性**: LynQ・Pazの専門能力をSpectraが適切に活用

#### ⚡ 効率性
- **無駄排除**: 不要な応答を事前に判定
- **最適化**: 最も適切なボットが応答
- **シーケンシャル**: LLM制約内での最適な処理

### 具体的動作例

#### 例1: Spectra直接応答
```
ユーザー: "今日はどんな調子？"
↓
Spectra: 深層分析（一般的な挨拶・雑談）
↓
Spectra: メタ判断（私が最適）
↓
Spectra: 直接応答（ファシリテーション）
```

#### 例2: 効率化された専門委譲
```
ユーザー: "このアルゴリズムの効率性を分析してください"
↓
Spectra: 深層分析（論理分析が必要）
↓
Spectra: メタ判断（LynQが最適）
↓
Spectra: 内部シグナル送信（中間Discord投稿なし）
↓
LynQ: シグナル受信 → 即座に専門分析応答
```

#### 例3: マルチエージェント協調
```
ユーザー: "プロジェクトを論理的に検証して、さらに創造的に発展させたい"
↓
Spectra: 深層分析（論理+創造の複合要求）
↓
Spectra: メタ判断（複数視点が必要）
↓
Spectra: 協調開始の調整投稿
↓
LynQ: 論理検証応答 → Paz: 創造発展応答 → Spectra: 統合調整応答
```

### 結論
この**基底エージェント設計**は：

1. **LLM知的活用** - 軽量判定ではなく高度な文脈理解・判断
2. **メタ認知実現** - 客観的な「誰が最適か」の判断
3. **独立性保持** - 各ボットの独立したDiscord Bot性質を維持
4. **価値最大化** - 各状況で最も価値ある応答を提供

**つまり、「LLMの知的判断力を最大活用」することで、高度で合理的な設計を実現**します。
